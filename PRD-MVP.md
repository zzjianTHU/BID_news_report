## 2. 产品背景

清华商业模式中心长期围绕不同研究方向持续产出内容与洞察，例如 AI、快消等方向。当前内容生产具备专业性和品牌背书，但在信息整合、结构化分发、用户订阅和持续触达方面仍有进一步产品化的空间。

本项目拟建设一个轻量化的信息站，支持多专题内容协作、自动抓取与人工上传并行、按用户可接受的阅读时间生成不同内容版本，并通过站内浏览与邮件推送的方式形成稳定的信息触达机制。

MVP 阶段不做付费验证，重点验证内容产品本身是否成立，验证团队是否能以较低运营成本稳定输出内容。

## 3. 产品愿景

打造一个由清华商业模式中心支持的、面向管理者与产业从业者的轻量化专题情报站，让用户在有限时间内高效获取更值得看的信息。

## 4. 产品定位

一个支持多专题协作、按阅读时长输出、以邮箱订阅驱动分发的轻量化情报平台。

对外表达建议：

“用更少时间，获取更值得看的 AI 与产业信息。”

## 5. MVP 核心目标

### 5.1 业务目标

1. 建立一个统一品牌下的内容分发平台，承载多个研究专题。
2. 跑通从信息采集、编辑筛选、内容生成到发布分发的完整链路。
3. 建立邮箱订阅机制，形成用户触达和回访习惯。
4. 在不引入复杂商业化能力的前提下，验证产品使用价值与内容生产效率。

### 5.2 用户目标

1. 用户可以按专题快速浏览当日值得关注的信息。
2. 用户可以按自己的时间预算选择阅读版本，例如 3 分钟版或 8 分钟版。
3. 用户可以通过邮箱稳定接收更新，而不是依赖主动访问站点。

### 5.3 团队目标

1. 不要求每个专题负责人每天撰写长文。
2. 通过“自动抓取 + 人工选择 + 系统生成初稿 + 轻编辑”的方式降低维护成本。
3. 支持不同专题由不同人员维护，但前台对用户呈现为统一平台体验。

## 6. 非目标

以下内容不纳入 MVP 范围：

1. 付费订阅与支付系统
2. 复杂会员等级体系
3. 企业版与团队版权限
4. 评论、社区、互动讨论
5. 复杂推荐算法
6. 多层审批流与重型 CMS
7. 深度个性化用户中心

## 7. 目标用户

### 7.1 核心用户

1. 关注 AI 与产业趋势的管理者
2. 创业者与产品负责人
3. 对商业模式、消费市场、行业变化有信息需求的研究者与从业者

### 7.2 次级用户

1. 对特定专题持续关注的学生
2. 希望低成本跟踪重点信息的助理型研究用户

### 7.3 用户共性需求

1. 时间有限，不愿自己大量筛选信息
2. 希望内容经过专业机构筛选和整理
3. 希望按兴趣专题订阅
4. 希望通过低门槛方式接收更新，例如邮箱

## 8. 核心价值主张

### 对用户

1. 节省信息筛选时间
2. 按专题获取重点内容
3. 按阅读时长获取适合自己的内容版本
4. 获得带有机构背书的信息整理与判断

### 对中心团队

1. 提升多专题内容协作效率
2. 将零散内容生产能力沉淀为统一产品
3. 通过统一分发渠道沉淀用户与订阅关系

## 9. 产品方案概述

产品采用“统一平台 + 多专题频道”的结构。

### 9.1 平台层

承载统一品牌与公共能力：

1. 首页
2. 今日更新入口
3. 阅读时长切换
4. 邮箱订阅
5. 方法说明
6. 机构背书

### 9.2 专题层

每个专题对应一个内容频道，由不同负责人维护。

首批建议专题：

1. AI
2. 快消

后续可扩展：

1. 商业模式创新
2. 消费科技
3. 产业观察

## 10. 用户场景

### 场景 1：快速浏览

用户打开首页，希望在 3 分钟内了解今天最值得看的内容，快速完成浏览并离开。

### 场景 2：专题订阅

用户只关心 AI 或快消，希望订阅对应专题，并通过邮件持续接收更新。

### 场景 3：中度阅读

用户有 8 分钟左右时间，希望看到比速览更完整的内容，包括背景和重点说明。

### 场景 4：持续回访

用户没有每天主动访问站点的习惯，但愿意通过邮箱定期接收内容并点击回站浏览。

## 11. MVP 功能范围

### 11.1 前台功能

1. 首页
2. 专题页
3. 每日简报页
4. 历史归档页
5. 方法/关于页
6. 邮箱订阅入口

### 11.2 内容呈现功能

1. 支持按专题浏览
2. 支持按日期查看每日简报
3. 支持 3 分钟版与 8 分钟版两种内容版本
4. 每条内容支持原文链接跳转

### 11.3 用户功能

1. 留邮箱订阅
2. 选择关注专题
3. 选择默认阅读时长偏好
4. 接收邮件推送

### 11.4 后台功能

1. 信息源管理
2. 手动上传资料
3. 候选内容池
4. 内容去重与基础标签分类
5. 编辑选择待发布内容
6. 生成不同阅读时长版本
7. 发布到站内
8. 推送邮件

## 12. 内容输入机制

### 12.1 自动抓取

支持从以下来源抓取内容：

1. RSS
2. 指定网页列表
3. 公开资讯站点
4. 研究机构发布页

### 12.2 手动上传

支持以下资料类型：

1. PDF
2. Word 或其他文档内容
3. 网页链接
4. 中心原创文章与内部资料

### 12.3 内容进入方式

所有内容统一进入候选池，供专题负责人筛选与编辑。

## 13. 内容生产工作流

### 13.1 角色定义

1. 管理员
2. 专题编辑/负责人

### 13.2 标准工作流

1. 系统按配置定时抓取信息源
2. 内容进入对应专题候选池
3. 专题负责人查看候选内容并筛选当天值得发布的条目
4. 系统为选中内容生成摘要初稿
5. 系统生成 3 分钟版与 8 分钟版内容编排
6. 专题负责人进行轻编辑，补充标题、标签、短评或修订内容
7. 发布为站内当日简报
8. 系统按订阅配置生成邮件并发送

### 13.3 工作流原则

1. 系统负责收集与初稿
2. 人负责判断与校订
3. 每个专题负责人只维护本专题内容
4. 尽可能降低人工写作成本

## 14. 内容输出结构

### 14.1 3 分钟版

适合快速浏览，突出最重要的信息：

1. 今日最重要的 3-5 条内容
2. 每条内容包含短摘要
3. 每条内容附带一句“为什么值得看”
4. 提供原文链接

### 14.2 8 分钟版

适合中度阅读，增加上下文和判断：

1. 更完整的摘要
2. 按主题或优先级分组
3. 增加背景说明
4. 增加中心视角的简短点评

2. 今日更新入口
3. 阅读时长切换入口
4. 专题入口
5. 邮箱订阅入口
6. 方法说明
7. 清华商业模式中心背书

### 15.2 专题页

模块建议：

1. 专题介绍
2. 今日简报
3. 历史归档
4. 订阅本专题

### 15.3 每日简报页

模块建议：

1. 日期
2. 专题信息
3. 阅读时长切换
4. 内容条目列表
5. 原文链接
6. 订阅入口

### 15.4 历史归档页

模块建议：

1. 按日期浏览
2. 按专题筛选
3. 基础搜索能力（可选，若开发资源不足可后置）

### 15.5 方法/关于页

模块建议：

1. 中心介绍
2. 信息来源说明
3. 内容生成机制
4. 编辑原则

## 16. 邮件订阅机制

### 16.1 订阅方式

用户通过邮箱完成订阅，可选择：

1. 关注专题
2. 默认阅读时长
3. 推送频率（建议 MVP 先支持每日或工作日）

### 16.2 邮件内容

邮件内容与站内简报保持一致，优先保证清晰和稳定，不在 MVP 阶段引入复杂个性化。

### 16.3 邮件目标

1. 形成稳定触达
2. 提升回访率
3. 建立用户关系资产

## 17. 权限与协作

### 17.1 管理员权限

1. 创建和管理专题
2. 配置信息源
3. 管理站点公共设置
4. 管理订阅与邮件策略

### 17.2 专题编辑权限

1. 查看本专题候选池
2. 上传本专题资料
3. 编辑本专题内容
4. 发布本专题简报

### 17.3 协作原则

1. 权限尽量简化
2. 优先按专题隔离，而不是按复杂角色层级隔离
3. 确保多人维护时前台体验仍统一

## 18. 成功指标

### 18.1 用户侧指标

1. 邮箱订阅转化率
2. 邮件打开率
3. 邮件点击率
4. 首页到专题页点击率
5. 3 分钟版与 8 分钟版使用占比

### 18.2 内容与运营指标

1. 每个专题的更新频率
2. 每次简报生产耗时
3. 每个专题每日成功发布率
4. 候选池到发布内容的转化效率

### 18.3 产品验证指标

1. 用户是否形成稳定订阅行为
2. 用户是否实际使用按时长阅读能力
3. 团队是否能轻量化维持日常运营

## 19. 风险与假设

### 19.1 关键假设

1. 用户认可“按时间预算阅读”的价值
2. 多专题内容能够在统一产品形态下成立
3. 团队能够以轻编辑方式维持稳定更新

### 19.2 主要风险

1. 内容维护节奏无法长期稳定
2. 自动抓取内容质量参差不齐，增加编辑负担
3. 3 分钟版与 8 分钟版差异不明显，用户感知弱
4. 邮件触达效果不如预期

## 20. 里程碑建议

### 阶段 1：产品定义与原型

1. 明确首批专题范围
2. 确认首页、专题页、简报页结构
3. 完成后台最小工作流定义

### 阶段 2：MVP 开发

1. 搭建内容模型
2. 完成前后台核心页面
3. 打通信息源配置、内容发布和邮件推送

### 阶段 3：内测验证

1. 先用 1-2 个专题试运行
2. 验证内容生产效率
3. 验证订阅和邮件指标

## 21. 第二阶段演进方向

MVP 验证完成后，可考虑逐步引入：

1. 深度版内容
2. 付费订阅
3. 更丰富的专题库与专题报告
4. 用户收藏与个人内容中心
5. 更细的推送偏好
6. 企业或团队版能力

## 22. 结论

本项目的本质不是建设传统内容网站，而是建设一个轻量化、可持续、多专题协作的信息产品平台。

MVP 阶段应以“跑通内容链路、建立订阅习惯、降低团队维护成本”为核心目标，避免过早引入复杂商业化与重型系统设计。
